# 012. Resource Indicators (RFC 8707) e Protected Resource Metadata (RFC 9728)

## 1. Contexto

A ADR-010 estabeleceu que o `aud` do access token deriva dos ResourceServers donos
dos Scopes pedidos, e que um ResourceServer **nao e requisitavel pelo parametro
`scope`** (apenas `Scope` e `IdentityScope` sao). Isso cobre bem APIs que expoem
scopes.

Porem, existe um cenario legitimo nao coberto: uma API/ResourceServer que **nao
trabalha com scopes** e faz a propria autorizacao por claims/roles do usuario,
precisando apenas que o token tenha o `aud` correto. Tambem ha o caso de um
client que quer indicar explicitamente em qual API usara o access token.

Forcar toda API a declarar um scope "coarse" (ex.: `api`) e paliativo, nao
solucao: nem toda API quer scopes. E pedir o RS pelo nome no `scope` reintroduziria
a conflacao `scope`/`aud` que a ADR-010 removeu.

O mecanismo padrao para isso e o **RFC 8707 (Resource Indicators for OAuth 2.0)**:
o parametro `resource` permite ao client indicar a URI da API alvo. O metadata
padronizado para descrever esses protected resources e o **RFC 9728 (OAuth 2.0
Protected Resource Metadata)**.

Avaliou-se reaproveitar a flag `ResourceServer.AllowScopeRequests` para liberar
`scope=<nome_do_RS>`. **Rejeitado:** criaria dois caminhos para obter `aud` pelo
parametro `scope`, com ambiguidade de precedencia, e reintroduziria a conflacao.

## 2. Decisao

Separar os eixos em mecanismos distintos, com compatibilidade para o caminho
legado de scopes:

- **Scopes ("o que posso fazer")**: parametro `scope`. A flag
  `ResourceServer.AllowScopeRequests` (default `true`) indica se os Scopes daquele
  ResourceServer podem ser pedidos via `scope`. Quando `false`, os scopes do RS
  nao sao requisitaveis por `scope`; a API pode ser alcancada apenas via
  `resource`.
- **Audience explicito ("onde o token vale")**: parametro `resource` do RFC 8707,
  resolvido contra protected resources cadastrados no realm.

Adicionar uma colecao de protected resources ao ResourceServer:

```csharp
public class ResourceServer
{
    public string Name { get; set; }
    public string? Audience { get; set; } // usado apenas no caminho por scope
    public ICollection<ProtectedResource> ProtectedResources { get; set; } = [];
    public ICollection<Scope> Scopes { get; set; } = [];
}

public class ProtectedResource
{
    public string ResourceUri { get; set; }
    public bool ShowInDiscoveryDocument { get; set; } = true;

    // RFC 9728 metadata
    public string? DisplayName { get; set; }
    public string? DocumentationUri { get; set; }
    public string? PolicyUri { get; set; }
    public string? TosUri { get; set; }
}
```

O nome de classe decidido e **`ProtectedResource`**. "Resource Indicator" fica
como termo do protocolo/request; `ProtectedResource.ResourceUri` e o valor que
casa com o parametro `resource`.

Regras:

- Um RS **nao** e requisitavel pelo nome no `scope` (mantem ADR-010). Pedir o
  nome do RS em `scope` resulta em `invalid_scope`.
- `ResourceServer.Audience` fica para compatibilidade e e usado **apenas** quando
  o access token e emitido por scopes sem `resource` explicito. Nesse caminho:
  `aud = ResourceServer.Audience ?? ResourceServer.Name`.
- Quando `resource` e informado, o `aud` emitido para aquele protected resource e
  o proprio `ProtectedResource.ResourceUri`. O `Audience` do RS nao e adicionado
  para esse mesmo RS nesse caminho.
- Um ResourceServer pode ter **varios** `ProtectedResource`s. Cada
  `ProtectedResource.ResourceUri` e unico por realm e pertence a exatamente um
  ResourceServer.
- O parametro `resource` e aceito no **authorize** endpoint e no **token**
  endpoint; pode repetir (multiplos resources), conforme o RFC 8707.
- `scope + resource` e aceito. Quando `resource` aparece, scopes de API so sao
  validos se o ResourceServer dono do scope estiver representado por pelo menos um
  dos `ProtectedResource`s pedidos; caso contrario, a combinacao e invalida e
  retorna `invalid_target`.
- **Autorizacao:** o client so pode indicar um `resource` cujo ResourceServer dono
  esteja em `AllowedResourceServers` (ou com `AllowAllResourceServers`). A
  permissao dos scopes continua seguindo ADR-011 (`AllowedScopes`,
  `AllowedResourceServers`, `AllowAllResourceServers`) e `AllowScopeRequests`.
- Um `resource` malformed, com fragmento, desconhecido, desabilitado ou nao
  aceitavel pela politica local retorna `invalid_target`.
- O token continua levando **apenas os scopes pedidos** (ADR-010); `resource`
  afeta o `aud` e o conjunto de ResourceServers/ProtectedResources efetivos.
- `AuthorizationCode` e refresh grants persistem os protected resources
  autorizados. No token endpoint, `resource` pode ser omitido ou ser subset do
  conjunto autorizado; fora disso, `invalid_target`.

## 3. Discovery e RFC 9728

A implementacao de Resource Indicators deve incluir Protected Resource Metadata
como parte do modelo padrao, nao apenas uma flag simples no discovery.

Regras de discovery:

- O metadata do Authorization Server publica `protected_resources` com os
  `ProtectedResource.ResourceUri` habilitados e publicos do realm.
- Para cada `ProtectedResource`, o RoyalIdentity deve conseguir gerar o documento
  RFC 9728 correspondente, contendo ao menos:
  - `resource`
  - `authorization_servers`
  - `scopes_supported`
  - `bearer_methods_supported`
  - `resource_name`
  - `resource_documentation`, `resource_policy_uri`, `resource_tos_uri` quando
    configurados
- Em producao, o local ideal para servir esse documento e o dominio da propria
  API/protected resource (por exemplo,
  `https://api.example.com/.well-known/oauth-protected-resource`). O RoyalIdentity
  pode gerar o JSON para que a API publique ou, em cenarios controlados, servir
  endpoints auxiliares por realm para administracao/desenvolvimento.
- Nao criar metadado custom como `resource_parameter_supported` como sinal
  principal. O padrao publico e `protected_resources` + metadata RFC 9728.

## 4. Consequencias

- `ResourceServer` ganha `ProtectedResources`; `ResourceServer.Audience` continua
  existindo para compatibilidade do caminho por scope.
- `RequestedResources`/`IWithResources`, contexts, handlers, validators,
  `AuthorizationCode`, refresh tokens e `DefaultTokenFactory` passam a carregar
  os resources pedidos/autorizados.
- O store de recursos precisa indexar `ProtectedResource.ResourceUri` por realm e
  validar unicidade.
- API audience-only deixa de precisar de scope "coarse"; o caminho padrao e
  `resource=<ProtectedResource.ResourceUri>`.
- Erro novo/ativado: `invalid_target` (RFC 8707).
- Implementado em fase propria do `.ai/plans/plan-resources-redesign.md`
  (**Fase 6: Resource Indicators e Protected Resource Metadata**), apos a Fase 5
  e antes do consentimento.
- Fora de escopo aqui: autorizacao fina da API, UMA/permissions/policies estilo
  Keycloak Authorization Services, introspection/reference tokens e SDK/middleware
  completo para Resource Servers hospedarem automaticamente o metadata RFC 9728.
