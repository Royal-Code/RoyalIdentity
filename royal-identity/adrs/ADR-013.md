# 013. Arquitetura modular e fronteiras (storages como facades)

> **Emenda (2026-06-17 — ADR-015):** o módulo de contas chama-se **`RoyalIdentity.UserAccounts`** (singular);
> a adaptação de um módulo rico às portas do IdP fica num **projeto de integração separado**
> (`{Module}.Integration`, análogo a `Storage.EntityFramework` para `Data.*`) — o **módulo puro não referencia o
> core**; os provedores de persistência usam a grafia **`.PostgreSql`** (`.Postgre` → `.PostgreSql`). Ver "§5 Emenda"
> e [ADR-015](ADR-015.md).

## 1. Contexto

O RoyalIdentity hoje é composto por `RoyalIdentity` (core: domínio, contratos,
contexts, handlers, endpoints, options, eventos e usuários), `RoyalIdentity.Pipelines`
(infra de pipeline, sem dependências), `RoyalIdentity.Storage.InMemory` (único storage),
`RoyalIdentity.Razor` (UI de conta) e `RoyalIdentity.Server` (host). O padrão
`IStorage.GetXStore(realm)` **já é**, na prática, uma *facade*: o core depende do
contrato e o `Storage.InMemory` o implementa.

O redesign de usuários (análise [an-users-arch.md](../.ai/analisys/an-users-arch.md),
que reescopa [an-users-final.md](../.ai/analisys/an-users-final.md)) expôs que o
modelo de usuário cresceu até virar, implicitamente, um **módulo de contas** sem
fronteiras desenhadas, e que faltam decisões estruturais para:

- **Persistência de produção** (EFCore/Postgres/Sqlite) por trás das *facades*, sem
  acoplar o core a um provedor.
- **Módulos de domínio** (contas de usuário, gestão de chaves) que têm domínio +
  persistência próprios e ciclo de vida independente do core.
- **Direção de dependência** entre core, dados, storages e módulos.

A implementação *in-memory* vinha sendo tratada implicitamente como destino. Na
verdade ela é uma **implementação fake/referência** (dev/test/integração/realm Demo),
não a forma de produção.

Esta ADR fixa a **arquitetura modular e as fronteiras**. As decisões concretas de
contratos da **borda de usuários + sessão** ficam na [ADR-014](ADR-014.md).

## 2. Decisão

### 2.1 Storages são facades; contratos no core

Os contratos/*facades* de storage ficam no **core `RoyalIdentity`**. Quem implementa
referencia o core (como o `Storage.InMemory` faz hoje). O core **não** depende de
nenhuma implementação de storage nem de módulo.

A implementação *in-memory* é **fake/referência** para dev/test/integração/demo —
não é a forma de produção. Ela permanece e ganha índices e seeds determinísticos,
mas não é o destino.

### 2.2 Projetos de dados puros (sem depender do core)

A persistência de produção é dividida em dois projetos de **dados puros**:

- **`RoyalIdentity.Data.Configuration`** — realms, clients, resources, keys, options.
- **`RoyalIdentity.Data.Operational`** — sessions, tokens, codes, consents.

A divisão segue **ciclo de vida e volume**: configuração é raramente escrita;
operacional é alta rotatividade, com TTL/cache. Esses projetos contêm `DbContext`,
entidades de persistência e queries; podem depender de EFCore, mas **não conhecem
tipos/contratos do core `RoyalIdentity`**.

### 2.3 Storage.EntityFramework adapta os dados às facades

**`RoyalIdentity.Storage.EntityFramework`** é o **único** projeto que adapta
`Data.Configuration`/`Data.Operational` para os contratos/*facades* do core. Os
mapeamentos e migrations por provedor ficam **sob** `Storage.EntityFramework.*`
(**`.PostgreSql`**, **`.Sqlite`**) — assim o nome dos projetos `Data.*` fica livre de
provedor. **`RoyalIdentity.Storage.Caching`** envolve as implementações de storage
com cache (relevante no operacional).

### 2.4 Módulos de domínio: domínio + persistência; API e UI fora

- **`RoyalIdentity.UserAccounts`** — módulo de contas de usuário (camada B):
  domínio rico, propriedades por escopo, credenciais, casos de uso administrativos,
  eventos, persistência própria.
- **`RoyalIdentity.KMS`** — vault de chaves/segredos/certificados: domínio +
  persistência próprios.

Regras dos módulos:

- Módulo = **domínio + persistência**. **API e UI são projetos separados** (um
  conjunto de projetos de API e de UI por módulo), **não** vivem dentro do módulo.
- O `UserAccounts` tem **persistência própria** e **não é adaptado** pelo
  `Storage.EntityFramework` do IdP; quando necessário, ele expõe sua própria
  implementação/integração para as portas de borda definidas na ADR-014
  (`ISubjectStore`, `ILocalUserAuthenticator`, `IUserClaimsProvider`).
- O módulo **nunca** vê tipos ricos do core (ex.: `IdentityScope`,
  `RequestedResources`); o seam de claims passa **apenas primitivos**
  (nomes de identity scopes + claim types). Ver ADR-014.

### 2.5 Sessão é da borda; persistência no operacional

O **contrato/serviço** de sessão (`IUserSessionStore`/`IUserSessionService`) fica no
**IdP** (core); a **persistência** de produção vai para `Data.Operational` (EFCore).
In-memory cobre dev/test. Detalhes do modelo e da semântica na ADR-014.

### 2.6 `RoyalIdentity.Abstractions` adiado

Extrair os contratos para um projeto `RoyalIdentity.Abstractions` seria uma mudança
estrutural grande e não é necessária agora: os contratos permanecem no **core** e os
projetos `Data.*` ficam puros sem precisar dele. Reavaliar apenas se um consumidor
externo precisar dos contratos sem arrastar o core.

## 3. Fronteiras de dependência (resumo)

```
RoyalIdentity.Pipelines            (base, sem dependências)
        ▲
RoyalIdentity  (core/IdP)          contratos/facades + modelo de borda + sessão + fluxo
        │  (facades implementadas por →)
        ├── Storage.InMemory                       (fake/referência; dev/test/integração/demo)
        ├── Storage.EntityFramework                (adapta Data.* às facades do core)
        │       └── .PostgreSql / .Sqlite          (mapeamentos/migrations por provedor)
        ├── Storage.Caching                        (cache sobre storage)
        │
        ├── Data.Configuration                     (dados puros, NÃO dependem do core)
        ├── Data.Operational                       (dados puros, NÃO dependem do core)
        │
        ├── UserAccounts                           (módulo: domínio + persistência própria)
        └── KMS                                     (módulo: domínio + persistência própria)
```

Invariantes de dependência:

- O **core não depende** de storages nem de módulos.
- **`Data.*` não depende do core** (só EFCore + tipos próprios).
- **Só `Storage.EntityFramework`** faz a ponte `Data.*` ↔ *facades* do core.
- Módulos têm **persistência própria**; **API/UI** são projetos separados.

## 4. Consequências

- A produção passa a ter persistência real (EFCore/Postgres) atrás das mesmas
  *facades* que a in-memory implementa; trocar in-memory→EFCore é registro de DI.
- A in-memory permanece como referência e pode ser substituída por Sqlite nos testes
  sem mudar contratos.
- A divisão `Data.Configuration`/`.Operational` permite políticas distintas de
  cache/TTL e evolução independente de schema por ciclo de vida.
- Cada módulo/fronteira tem **ADR e plano próprios**, executados em ordem:
  - **borda + sessão** (camadas A+C) — [ADR-014](ADR-014.md) +
    [plan-users-edge-session.md](../.ai/plans/plan-users-edge-session.md) (imediato);
  - **persistência de dados** — `plan-data-persistence` (futuro);
  - **módulo de contas** `UserAccounts` — ADR-015 + `plan-users-accounts-module-v2` (futuro);
  - **KMS** — ADR + `plan-kms` (futuro).
  Os três últimos estão no [backlog-001.md](../.ai/backlogs/backlog-001.md).
- Esta ADR é de **escopo/arquitetura**: não introduz código. Os contratos concretos
  da borda de usuários e a semântica de sessão estão na ADR-014.
- Anti-YAGNI preservado: as *costuras* (facades) são desenhadas agora; os módulos e a
  persistência EFCore são construídos depois, sem reescrever a borda.

## 5. Emenda (2026-06-17 — ADR-015 / módulo `UserAccounts`)

Refinamentos das decisões §2.3/§2.4 ao detalhar o primeiro módulo rico:

- **Nome:** `RoyalIdentity.UserAccounts` (singular), não `UsersAccounts`.
- **Projeto de integração separado:** um módulo rico vira uma **família** — módulo **puro**
  (`RoyalIdentity.UserAccounts`, **sem** referência ao core), **`.Integration`** (referencia core + módulo; implementa
  as portas de borda `ISubjectStore`/`ILocalUserAuthenticator`/`IUserClaimsProvider`/`IUserDirectory`), e provedores
  **`.PostgreSql`**/**`.Sqlite`**. Esse padrão de **adapter** é a forma preferida (espelha `Storage.EntityFramework`
  adaptando `Data.*`) e mantém a **API/UI administrativa livre do IdP**.
- **Grafia de provedor:** padronizar **`.PostgreSql`** em toda a solução (inclui o futuro
  `Storage.EntityFramework.Postgre` → `.PostgreSql`).
- Detalhes do módulo: [ADR-015](ADR-015.md).
