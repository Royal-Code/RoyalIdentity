# 006. Convenção de constantes e uso de constantes externas

## 1. Contexto

O código herdado do IdentityServer4 e a reestruturação do RoyalIdentity trouxeram diferentes classes de constantes, como `OidcConstants`, `JwtClaimTypes` e `ServerConstants`.

Parte dessas constantes representa nomes padronizados de OIDC, OAuth2 e JWT. Outra parte representa nomes específicos do RoyalIdentity.

Manter constantes próprias para valores já definidos por pacotes oficiais aumenta duplicidade, risco de divergência semântica e dificuldade de manutenção.

## 2. Decisão

As constantes específicas do RoyalIdentity devem ser consolidadas na hierarquia `Constants.*`.

Quando uma constante representar um valor padronizado e já existir em pacote oficial usado pelo projeto, como os pacotes Microsoft de OpenID Connect, JWT ou IdentityModel, o código deve preferir a constante externa.

Constantes próprias só devem ser mantidas quando:
- O valor for específico do RoyalIdentity;
- O pacote externo não oferecer uma constante equivalente;
- A constante externa existir, mas não representar exatamente a mesma semântica.

Classes top-level legadas de constantes devem ser removidas após a migração dos callers.

## 3. Consequências

O código fica mais alinhado aos padrões oficiais de OIDC, OAuth2 e JWT.

Novas constantes devem ser avaliadas antes de serem adicionadas ao projeto.

A hierarquia `Constants.*` passa a representar apenas convenções próprias ou agrupamentos necessários do RoyalIdentity.

Refatorações desse tipo devem ser guiadas por build e testes, pois o uso de `global using static` pode esconder referências não qualificadas.
