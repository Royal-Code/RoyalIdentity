# 016. Biblioteca técnica compartilhada `RoyalIdentity.Security`

## 1. Contexto

O ADR-013 define a arquitetura modular do produto e a convenção de nomes `RoyalIdentity.*` para o core, os módulos de domínio e suas famílias (`.Integration`, `.PostgreSql`, `.Sqlite`).

Primitivas técnicas genéricas — random criptográfico, encoding Base64Url, hashing, comparação em tempo constante, password hashing e material de chaves — hoje vivem no core `RoyalIdentity` em `Utils/`, `Extensions/` e `Models/Keys/`. Isso gera duplicação: o módulo puro `RoyalIdentity.UserAccounts`, que por contrato não referencia o core, precisou recriar `DefaultSubjectIdGenerator`; e o futuro `RoyalIdentity.KMS` precisará manipular chaves sem depender do IdP.

Essas primitivas não conhecem realm, client, scope, token, store nem pipeline. Não são domínio do IdP. Colocá-las no core força um dilema: ou o consumidor referencia o core (quebrando a regra de módulo puro do ADR-013), ou cada módulo duplica a primitiva.

O backlog registrava esse trabalho sob o nome `RoyalIdentity.Security`. Cogitou-se posicioná-lo sob uma raiz de namespace distinta `RoyalCode.*` para sinalizar que não é domínio do IdP. Essa raiz, porém, **já está ocupada**: `RoyalCode.*` é o ecossistema externo de bibliotecas de produtividade consumido pelo projeto (`RoyalCode.SmartCommands`, `RoyalCode.Entities`, `RoyalCode.Aggregates`, `RoyalCode.WorkContext`, `RoyalCode.SmartProblems`, `RoyalCode.SmartSearch`, etc.). Criar um projeto interno do produto sob `RoyalCode.*` colidiria conceitualmente com esses pacotes de terceiros, confundindo o que é biblioteca externa com o que é código do produto.

## 2. Decisão

Introduzir uma **biblioteca técnica compartilhada** chamada `RoyalIdentity.Security`, dentro do namespace do produto `RoyalIdentity.*`, para componentes genéricos reutilizáveis que não pertencem ao domínio do IdP.

O nome usa a raiz `RoyalIdentity` — e **não** a raiz externa `RoyalCode` — para manter o código do produto separado do ecossistema externo `RoyalCode.*` e evitar colisão de namespaces. O caráter "não-domínio" do projeto é expresso pela sua categoria (biblioteca técnica de folha) e pelas suas regras de dependência, **não** por uma raiz de namespace própria.

`RoyalIdentity.Security` é uma biblioteca de folha: não é o core, não é um módulo de domínio nem família de módulo. Regras:

- Não referencia o core `RoyalIdentity` nem nenhum módulo de domínio (`RoyalIdentity.UserAccounts`), storage, Razor ou Server.
- Não referencia `Microsoft.AspNetCore*` (remove o `FrameworkReference` global, como o módulo puro já faz).
- Expõe implementações concretas reutilizáveis, não abstrações de borda compartilhadas (sem `IPasswordHasher`, `IKeyManager`, stores etc.).
- Sua API é `public` e estável o suficiente para ser empacotável no futuro, mas a publicação NuGet externa não é exigida agora.
- O core `RoyalIdentity`, o storage e os módulos puros (`RoyalIdentity.UserAccounts`) podem referenciar `RoyalIdentity.Security`.

A escolha de algoritmo por realm/client/resource, o vocabulário OIDC (`at_hash`, `c_hash`, PKCE, `ClientSecret`, `PrivateKeyJwt`) e a integração com ASP.NET DataProtection permanecem no core e ficam fora desta biblioteca.

Esta decisão emenda o ADR-013: além de core, módulos de domínio e famílias, a convenção de nomes `RoyalIdentity.*` passa a admitir **bibliotecas técnicas de folha** (a primeira é `RoyalIdentity.Security`), que um módulo puro pode referenciar sem violar a regra de "sem dependência do core".

## 3. Consequências

Como a fronteira de módulo puro não pode mais ser expressa apenas por prefixo de namespace (`RoyalIdentity.Security` é `RoyalIdentity.*` e ainda assim pode ser referenciado por um módulo puro), os testes de arquitetura (`Tests.Architecture`) passam a impor a regra **por projeto específico**: um módulo puro pode referenciar `RoyalIdentity.Security`, mas continua proibido de referenciar o core `RoyalIdentity` ou outros módulos de domínio.

Os testes de arquitetura também impõem as fronteiras de `RoyalIdentity.Security`: nenhuma referência ao core `RoyalIdentity`, a módulos de domínio nem a `Microsoft.AspNetCore*`.

Componentes técnicos hoje duplicados ou presos ao core podem ser consumidos por core, storage, módulos puros e pelo futuro KMS a partir de uma única fonte, sem violar a regra de módulo puro do ADR-013.

Mantém-se a clareza contínua sobre o que é "técnico genérico" (vai para `RoyalIdentity.Security`) versus "domínio/borda do IdP" (permanece no core ou em módulos). Componentes com semântica de protocolo não devem migrar.

Futuros componentes técnicos genéricos compartilhados (ex.: utilitários consumidos pelo KMS) devem entrar em `RoyalIdentity.Security` (ou em nova biblioteca técnica de folha sob `RoyalIdentity.*`) em vez de serem duplicados nos módulos. A raiz externa `RoyalCode.*` permanece reservada às bibliotecas de produtividade de terceiros e não recebe código do produto.
