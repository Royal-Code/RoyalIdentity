# 016. Camada técnica compartilhada `RoyalCode.*`

## 1. Contexto

O ADR-013 define a arquitetura modular do produto e a convenção de nomes `RoyalIdentity.*` para o core, os módulos de domínio e suas famílias (`.Integration`, `.PostgreSql`, `.Sqlite`).

Primitivas técnicas genéricas — random criptográfico, encoding Base64Url, hashing, comparação em tempo constante, password hashing e material de chaves — hoje vivem no core `RoyalIdentity` em `Utils/`, `Extensions/` e `Models/Keys/`. Isso gera duplicação: o módulo puro `RoyalIdentity.UserAccounts`, que por contrato não referencia o core, precisou recriar `DefaultSubjectIdGenerator`; e o futuro `RoyalIdentity.KMS` precisará manipular chaves sem depender do IdP.

Essas primitivas não conhecem realm, client, scope, token, store nem pipeline. Não são domínio do IdP. Colocá-las em qualquer projeto `RoyalIdentity.*` força um dilema: ou o consumidor referencia o core (quebrando a regra de módulo puro do ADR-013), ou cada módulo duplica a primitiva.

O backlog registrava esse trabalho sob o nome `RoyalIdentity.Security`.

## 2. Decisão

Introduzir um tier de bibliotecas técnicas compartilhadas sob o namespace raiz `RoyalCode.*`, distinto de `RoyalIdentity.*`, para componentes genéricos reutilizáveis que não pertencem ao domínio do IdP.

O primeiro projeto desse tier é `RoyalCode.Security`. O nome usa a raiz `RoyalCode` — e não `RoyalIdentity.Security` como no backlog — precisamente para sinalizar que não é um módulo de domínio do IdP, e sim uma biblioteca técnica que o IdP consome.

Regras do tier `RoyalCode.*`:

- Não referencia nenhum projeto `RoyalIdentity.*`.
- Não referencia `Microsoft.AspNetCore*` (remove o `FrameworkReference` global, como o módulo puro já faz).
- Expõe implementações concretas reutilizáveis, não abstrações de borda compartilhadas (sem `IPasswordHasher`, `IKeyManager`, stores etc.).
- Sua API é `public` e estável o suficiente para ser empacotável no futuro, mas a publicação NuGet externa não é exigida agora.
- O core `RoyalIdentity`, o storage e os módulos puros (`RoyalIdentity.UserAccounts`) podem referenciar `RoyalCode.*`.

A escolha de algoritmo por realm/client/resource, o vocabulário OIDC (`at_hash`, `c_hash`, PKCE, `ClientSecret`, `PrivateKeyJwt`) e a integração com ASP.NET DataProtection permanecem no core e ficam fora deste tier.

Esta decisão emenda o ADR-013: a convenção de nomes do produto passa a admitir, além de `RoyalIdentity.*`, o tier técnico `RoyalCode.*`.

## 3. Consequências

Os testes de arquitetura (`Tests.Architecture`) passam a impor também as fronteiras de `RoyalCode.*`: nenhuma referência a `RoyalIdentity*` nem a `Microsoft.AspNetCore*`.

Componentes técnicos hoje duplicados ou presos ao core podem ser consumidos por core, storage, módulos puros e pelo futuro KMS a partir de uma única fonte, sem violar a regra de módulo puro do ADR-013.

A introdução de uma raiz de namespace adicional exige clareza contínua sobre o que é "técnico genérico" (vai para `RoyalCode.*`) versus "domínio/borda do IdP" (permanece em `RoyalIdentity.*`). Componentes com semântica de protocolo não devem migrar.

Futuros componentes técnicos genéricos compartilhados (ex.: utilitários consumidos pelo KMS) devem entrar no tier `RoyalCode.*` em vez de serem duplicados nos módulos.
