# 014. Redesign da borda de usuários e sessão (refina ADR-005)

> **Emenda (2026-06-17 — ADR-015 / módulo `UserAccounts`):** o seam de claims muda de
> `IUserPropertyProvider` (retornando `UserClaimDto[]`) para **`IUserClaimsProvider`** retornando
> **`IReadOnlyList<Claim>`** (`System.Security.Claims.Claim`); `UserClaimDto` é **removido**; o getter do gateway
> passa de `IUserDirectory.GetPropertyProvider` para **`GetClaimsProvider`**. Detalhes na seção "§4 Emenda" ao final
> e na [ADR-015](ADR-015.md). Implementação na **Fase 2** de
> [plan-users-accounts-module-v2.md](../.ai/plans/plan-users-accounts-module-v2.md).

## 1. Contexto

A [ADR-005](ADR-005.md) decidiu que o RoyalIdentity teria gerenciamento de usuários
próprio, configurável por realm, em vez de estender o ASP.NET Core Identity. A
implementação atual cumpriu o objetivo, mas misturou responsabilidades na **borda** do
IdP:

- `IdentityUser`/`DefaultIdentityUser` é um objeto **rico** que carrega serviços e
  monta o próprio `ClaimsPrincipal`; `UserDetails` é um POCO paralelo.
- `IUserStore` e `IUserDetailsStore` são a **mesma** classe; a resolução de login
  (username/email) vive no store.
- `IdentitySession` guarda o **usuário vivo**; `UserSessionStore` lê `HttpContext`.
- `sub` = `Username` (identificador protocolar acoplado ao login).
- **Lockout** existe em 3 lugares; a **sessão** é criada como **efeito colateral** da
  verificação de senha.
- "Ativo" é inconsistente em 3 caminhos (cookie, token endpoint, authorize).
- `LoginPageService` (Razor) carrega a **máquina de estados** do login; o
  `IdentityRevalidatingAuthenticationStateProvider` está quebrado.

A [ADR-013](ADR-013.md) fixou a arquitetura modular: a borda é feita de **facades**;
o **modelo rico de contas** migra para o módulo `RoyalIdentity.UserAccounts` (futuro);
*storages* são facades e a in-memory é fake/referência. Esta ADR decide o **redesign da
borda de usuários + sessão dentro do IdP** (camadas A + C), atrás das facades, mantendo
a in-memory como backing. É **refinamento** da ADR-005 (não a nega): a gestão própria de
usuários por realm permanece — agora por **composição** e, no futuro, pelo módulo.

## 2. Decisão

### 2.1 Composição sobre herança; modelo de borda enxuto

O objeto rico `IdentityUser` é substituído por **contratos de borda** (facades) + um
**modelo enxuto**. A borda **não** guarda o agregado de conta; usa `Subject` +
`IProfileService`. O modelo rico (perfil, credencial, security state, claims,
identidades externas) **não pertence** à biblioteca IdP — vai para o módulo
`UserAccounts` (ADR-013).

### 2.2 `SubjectId` imutável, separado de `Username`

`sub` = **`SubjectId`**, um identificador **estável e imutável**, separado de
`Username`/email. O `SubjectId` é a chave de lookup e **não deriva** do username.
Qualquer implementação de store **rejeita** alterar o `SubjectId` de uma conta
existente. Trocar o `Username` **não** muda o `sub`.

### 2.3 Contratos de borda (no core `RoyalIdentity`)

| Contrato | Responsabilidade exata | NÃO faz |
|---|---|---|
| `ISubjectStore` | `FindBySubjectId(sub) → Subject?`; `IsActive(sub)`. Lookup mínimo do sujeito. | Não devolve agregado rico; não resolve login; não modela email. |
| `ILocalUserAuthenticator` | `AuthenticateLocal(login, password) → AuthenticationResult`. Resolve identificador (username/email/fictício), **verifica credencial** e aplica **lockout**. | Não inicia sessão; não escreve cookie; não decide prompt/consent. |
| `IProfileService` (mantido) | Orquestra claims de token/userinfo; chama `IUserClaimsProvider` com **primitivos**; "ativo" combinado. | Não conhece o modelo rico de conta. |
| `IUserClaimsProvider` | `GetClaims(sub, identityScopeNames, claimTypes) → IReadOnlyList<Claim>`. Projeta propriedades→claims. | Não recebe `IdentityScope`/`RequestedResources` (só strings). |
| `IUserSessionStore` (puro) | `Create/FindById/RecordClient/End`. Persistência de sessão. | Sem `HttpContext`; sem "current". |
| `IUserSessionService` | current, `IsSessionValid(principal)`, start/end, record client. | — |
| `ISubjectPrincipalFactory` | `Create(subject, session) → ClaimsPrincipal` mínimo. | **Não emite roles/claims de perfil.** |
| `LoginFlowService` | Orquestra o login → **flow-result (enum)**. | Sem regra de conta; sem render. |
| `IUserDirectory` (gateway) | Getters realm-bound das portas de conta. | Não guarda dados do IdP. |
| `IAuthorizationContextResolver` | `Resolve(returnUrl) → AuthorizationContext?` para login/consent/logout. | — |
| `ICurrentRealmAccessor` | Fornece o realm corrente para serviços de orquestração. | **Só resolve realm**; não faz I/O de cookie. |

Tipos de borda (mínimos, no core): `Subject(SubjectId, DisplayName, IsActive)`;
`AuthenticationResult(Success | Reason{NotFound|Inactive|InvalidCredentials|Blocked} + Subject)`;
`UserSession(...)`; `UserSessionClient(ClientId, FirstSeenAt, LastSeenAt)`.

### 2.4 Gateway de contas e resolver de contexto (nomes fechados)

Os nomes provisórios da análise/plano ficam **fechados** assim:

- **`IUserDirectory`** (gateway de contas, no core) — expõe as portas de conta com
  getters **realm-bound**: `GetSubjectStore(realm)`, `GetLocalAuthenticator(realm)`,
  `GetClaimsProvider(realm)`. Implementado in-memory agora; pelo módulo
  `UserAccounts` depois. O `IStorage` fica **só com dados do IdP** (incluindo o store
  de sessão via `GetUserSessionStore(realm)`). Trocar in-memory→módulo é registro de DI.
- **`IAuthorizationContextResolver`** (no core) — recebe o `GetAuthorizationContextAsync`
  que hoje vive no `ISignInManager`. Usado por login/consent/logout; `SessionContextService`
  (Razor) **delega** a ele. O `ISignInManager` **não** mantém esse método.
- **`ICurrentRealmAccessor`** (no core) — fornece o realm corrente para serviços de
  orquestração, isolando o core do `HttpContext`. A implementação HTTP lê
  `HttpContext.GetCurrentRealm()`; testes usam um fake.

> Se durante a implementação um nome melhor surgir, ele substitui o aqui fechado
> **dentro desta ADR** (atualizar a ADR), não por divergência silenciosa no código.

### 2.5 Regra de resolução de realm: `Realm` nunca é parâmetro de método

Decisão do autor, registrada aqui como regra de design:

- **Portas realm-bound** (`ISubjectStore`, `ILocalUserAuthenticator`,
  `IUserClaimsProvider`, `IUserSessionStore`) recebem o realm **na construção**
  (a fábrica/gateway liga o realm; os métodos **não** recebem realm; **não** leem
  `HttpContext`). É o padrão `GetXStore(realm)` já existente.
- **Serviços de orquestração** (`IUserSessionService`, `LoginFlowService`,
  `DefaultSignOutManager`) resolvem o realm do **ambiente** via `ICurrentRealmAccessor`,
  ou de um argumento que **já carrega** realm (ex.: `client.Realm`).
- O **`ICurrentRealmAccessor` cobre apenas a resolução de realm**. O **I/O de cookie**
  (sign-in/sign-out) continua via `HttpContext` — o accessor **não** é uma camada de
  abstração de tudo que é HTTP. Isso evita ler "core sem `HttpContext`" como regra
  absoluta: a borda HTTP segue usando `HttpContext` para o que é HTTP por natureza.
- `ISubjectPrincipalFactory` **não** precisa de realm (lê de `subject`/`session`).

> Vantagem: a fábrica aceita realm explícito, então as portas funcionam também
> **fora de request** (ex.: jobs futuros de GC de sessão), sem depender de `HttpContext`.

### 2.6 Sessão (camada C): serializável, store puro, current num serviço

- `UserSession` é **serializável**: guarda `SubjectId` (não o usuário vivo), `sid`,
  `realm`, `idp`, `amr`, `auth_time`, `clients`, estado ativo. `UserSessionClient` tem
  **igualdade por `ClientId`** (dedup) — nunca uma lista sem dedup.
- `IUserSessionStore` é **puro**: sem `HttpContext`, sem "current"; realm-bound via
  `IStorage.GetUserSessionStore(realm)`.
- `IUserSessionService` provê **current**, validação, start/end e record-client,
  resolvendo o realm por `ICurrentRealmAccessor` (sem realm em método).
- **Sessão deixa de ser efeito colateral** da verificação de senha: é criada **apenas
  após autenticação bem-sucedida**, no sign-in, pelo `LoginFlowService`.

### 2.7 "Ativo" unificado; sessão ausente ⇒ inválida

Uma única regra, usada nos 3 caminhos (cookie, token endpoint, authorize):

```
ativo := ISubjectStore.IsActive(sub)
         && (principal tem sid ? IUserSessionService.IsSessionValid(principal) : true)
```

**Sessão ausente com `sid` no principal ⇒ inválida.** `IProfileService.IsActiveAsync`
implementa a combinação; `ActiveUserValidator` e `PromptLoginDecorator` se alinham a ela.

### 2.8 Principal mínimo de sessão; trade-off de roles/claims

`ISubjectPrincipalFactory` monta o principal **mínimo** de sessão a partir de
`Subject` + `UserSession`: **`sub`, `name`, `auth_time`, `sid`, `idp`, `amr`**.

**Roles e claims de perfil NÃO entram no cookie/principal de sessão por esta factory.**
Elas saem por `IProfileService`/`IUserClaimsProvider` (token/userinfo).

> **Trade-off (aceito):** roles/claims **deixam o cookie**. Isso **não** afeta
> token/userinfo (que usam o `IProfileService`). A consequência é que uma **futura UI
> admin** com `[Authorize(Roles = ...)]` **não** poderá ler roles do cookie de sessão e
> precisará de **mecanismo próprio** (claims transformation no pipeline da UI, ou um
> principal específico do realm admin). Decisão consciente: o principal de sessão fica
> enxuto e protocolar; autorização de UI é um problema separado.

### 2.9 Seam de claims: só primitivos cruzam a fronteira

`IProfileService` permanece no core e chama
`IUserClaimsProvider.GetClaims(sub, identityScopeNames, claimTypes)` passando
**apenas primitivos** (nomes de identity scopes + claim types) e recebendo
`IReadOnlyList<Claim>` (`System.Security.Claims.Claim`, tipo da BCL). O provider **nunca** vê
`IdentityScope`/`RequestedResources` — isso mantém `UserAccounts` independente do core.

### 2.10 Login como cola; erro genérico preservado

`LoginFlowService` (no core) orquestra:
`ILocalUserAuthenticator` → `IUserSessionService.Start(subject, method)` →
`ISubjectPrincipalFactory` → cookie sign-in → **flow-result (enum)** (consent /
signed-in / callback / profile / error). `LoginPageService` (Razor) vira **adaptador**:
monta view model e traduz o flow-result em redirect/render. O **erro de login é genérico
por default** (anti-enumeração); o **motivo interno** é preservado para evento/auditoria.

### 2.11 Remoções

São removidos: `IdentityUser`/`DefaultIdentityUser`, `UserDetails`, `IUserStore`,
`IUserDetailsStore`, `IdentitySession`, `ValidateCredentialsResult`,
`CredentialsValidationResult`, `IdentityRevalidatingAuthenticationStateProvider`.

## 3. Consequências

- A borda fica **100% facade**: contratos no core, in-memory como referência, módulo
  `UserAccounts` encaixando depois **sem reescrever a borda**.
- `sub` passa a ser `SubjectId` (≠ username): **testes que assertam `sub == username`
  serão atualizados** para os `SubjectId` determinísticos dos seeds.
- Lockout passa a existir em **um lugar só** (no autenticador / `LockoutPolicy`).
- A sessão deixa de carregar objeto rico e o store deixa de tocar `HttpContext`;
  logout SSO (front/back-channel) passa a operar por `SubjectId`.
- A UI deixa de conhecer `Subject`/`UserSession`/`ClaimsPrincipal`/stores/cookie; o
  roteamento de login vive no `LoginFlowService`.
- O **trade-off de roles fora do cookie** (§2.8) é registrado como dívida conhecida
  para a futura UI admin.
- `SecurityStamp` (invalidação de cookie/sessão) fica **reservado** (só campo, se útil),
  sem regra ligada neste escopo.
- A ADR-005 é **refinada**: gestão própria de usuários por realm permanece, agora via
  composição + módulo `UserAccounts`.
- Implementado por [plan-users-edge-session.md](../.ai/plans/plan-users-edge-session.md)
  (9 fases; in-memory como backing). O módulo `UserAccounts` e a persistência EFCore
  são trabalho posterior, com ADR e plano próprios (ADR-013 + backlog).
- Fora de escopo aqui: o domínio rico de contas (emails opcional/múltiplo/fictício, ID
  externo, propriedades por escopo no módulo, eventos, inbox/outbox, replicação),
  login externo/MFA/passwordless (apenas a **costura** extensível é preparada) e a
  persistência EFCore.

## 4. Emenda (2026-06-17 — módulo `UserAccounts` / ADR-015)

Ao detalhar o módulo de contas, o seam de claims (§2.3/§2.9) foi revisado. Mudanças **decididas**
(implementadas na **Fase 2** de [plan-users-accounts-module-v2.md](../.ai/plans/plan-users-accounts-module-v2.md)):

- **`IUserPropertyProvider` → `IUserClaimsProvider`**; **`IUserDirectory.GetPropertyProvider` → `GetClaimsProvider`**;
  **`MemoryUserPropertyProvider` → `MemoryUserClaimsProvider`**.
- O retorno passa de `UserClaimDto[]` para **`IReadOnlyList<Claim>`**; **`UserClaimDto` é removido**.
- **Motivo:** o consumidor imediato (o IdP) já usa `Claim` para emitir token/userinfo, e `Claim` é tipo da **BCL**,
  não do core — não havia ganho de desacoplamento no DTO de retorno. O desacoplamento real vem de **não** passar
  `IdentityScope`/`RequestedResources`: os **parâmetros** seguem primitivos (nomes de scopes + claim types).
- **Limite:** `Claim` aparece **só** na borda do IdP e no projeto `.Integration` do módulo; o domínio/persistência do
  módulo fala em conta/email/role/propriedade/definição/valor.
- A regra "Realm nunca é parâmetro de método" (§2.5) é preservada; no módulo, **só o `.Integration` conhece `Realm`**
  (traduz para `RealmId` + options do módulo).
